## Custom Project
### Dataset
The domain for this project is retail sales analytics, using two related tables loaded from CSV files:
- stores — store metadata (store_id, store_name, city, region)
- sales — individual transactions (sale_id, store_id, product_category, quantity, amount, sale_date)
These tables form a 1‑to‑many relationship:
- One store can appear in many sales
- The relationship is defined by the shared key store_id
This structure supports KPI analysis such as revenue by region, category performance, and store‑level ranking.

## Phase 5 Custom Project
I built a complete custom analytics pipeline using both SQLite and DuckDB.
This pipeline extends the example by adding new KPIs, new SQL scripts, a unified KPI table, and an export table.
### KPI Tables and Their Purpose
Each KPI table was designed to answer a specific business question.
Here’s what each one does and why it matters:
##### 1. Revenue by Region (kpi_revenue_by_region)
Measures: Total revenue, Sale count, Average sale amount, Grouped by region
Shows which geographic areas drive the business. Helps with staffing, inventory, and marketing allocation.
#### 2. Sales Aggregate (kpi_sales_aggregate)
Measures: Total number of sales, Total revenue, Average sale amount
Provides a high‑level snapshot of business performance. Useful for dashboards and trend analysis.
#### 3. Sales by Category (kpi_sales_by_category)
Measures: Revenue by product category, Sale count, Average sale amount
Identifies which categories are profitable and which may need promotion or discontinuation.
#### 4. Revenue by Store (kpi_revenue_by_store)
Measures: Revenue per store, Sale count, Average sale amount, Region and city context
Shows which stores outperform others. Helps with resource allocation and performance evaluation.
#### 5. Top‑Selling Product Categories
Measures: Highest‑revenue categories, Ranked in descending order
Highlights the categories that drive the majority of revenue. Useful for merchandising and marketing.
#### 6. Store Performance Ranking (kpi_store_performance_ranking)
Measures: Store ranking based on revenue and average sale amount
Provides a competitive view of store performance. Helps leadership identify top performers.
#### 7. Materialized KPI Table (jt_kpi_all)
- Combines all KPI tables into a single analytics‑ready dataset
- Simplifies BI dashboards and reporting
- Ensures all KPIs are aligned and consistent
#### 8. Export Table (export_kpi_all)
- Creates a clean, flattened table for external tools
- Ideal for Excel, Tableau, Power BI, or CSV export
Debugging scripts were critical to ensuring the pipeline worked correctly across both SQLite and DuckDB.
Debug scripts included
`jt_debug_sale_count.sql`, `jt_debug_store_count.sql`,`jt_debug_sale_peek.sql`, `jt_debug_store_peek.sql`, `jt_debug_data.sql`
These Verify that
- Dataset is loaded correctly Before running KPIs
- Catch early pipeline failures and identify: Missing tables, Incorrect column names, CSV load issues, Schema mismatches
- Prevent cascading errors
If the base tables were wrong, every KPI would fail.
Debugging ensured the foundation was solid.
- Support multi‑engine consistency
SQLite and DuckDB behave differently.
Debug scripts ensured both engines produced identical results.
